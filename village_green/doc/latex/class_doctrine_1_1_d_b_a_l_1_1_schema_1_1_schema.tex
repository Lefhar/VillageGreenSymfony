\doxysubsection{Detailed Description}
Object representation of a database schema.

Different vendors have very inconsistent naming with regard to the concept of a \char`\"{}schema\char`\"{}. Doctrine understands a schema as the entity that conceptually wraps a set of database objects such as tables, sequences, indexes and foreign keys that belong to each other into a namespace. A Doctrine \mbox{\hyperlink{class_doctrine_1_1_d_b_a_l_1_1_schema_1_1_schema}{Schema}} has nothing to do with the \char`\"{}\+SCHEMA\char`\"{} defined as in Postgre\+SQL, it is more related to the concept of \char`\"{}\+DATABASE\char`\"{} that exists in My\+SQL and Postgre\+SQL.

Every asset in the doctrine schema has a name. A name consists of either a namespace.\+local name pair or just a local unqualified name.

The abstraction layer that covers a Postgre\+SQL schema is the namespace of an database object (asset). A schema can have a name, which will be used as default namespace for the unqualified database objects that are created in the schema.

In the case of My\+SQL where cross-\/database queries are allowed this leads to databases being \char`\"{}misinterpreted\char`\"{} as namespaces. This is intentional, however the CREATE/\+DROP SQL visitors will just filter this queries and do not execute them. Only the queries for the currently connected database are executed.
